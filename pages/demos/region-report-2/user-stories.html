MiTrials Phase 3 — Analysis UI

Target: June 1 · Scope: Unified Analysis UI (trials + region reports)

Problem: Trial data and region-based analysis live in separate workflows, which limits insight and makes trends harder to interpret. Goal: One Analysis UI for time-series analysis, flexible regions (protocol + ad hoc), exclusions, and annotations — without mutating raw data.

Actors

  • Agronomist
  • Field Scientist
  • QC Team Member
  • Commercial / Sales (read-only or summary)

Entry points

  • Trial Detail → Analyze CTA → Analysis UI
  • Region Report → Open in Analysis UI → bridge copy (see sidebar link)
  • Field List → trial → Analyze Performance
  • Deep link URL → breadcrumb + share

Triggers

Open Analysis UI · select time range / dataset · apply exclusions · create annotation · switch NDVI / Yield layers.

Mapped to toolbar, map tools, and layer control on analysis-ui.html.

Core entities (reference)

Backend contract — UI reflects Trial, Region (protocol_assigned | ad_hoc), TimeSeries, ExclusionZone, Annotation, AnalysisState.

Trial · Region · TimeSeries(ndvi, yield) · ExclusionZone · Annotation · AnalysisState

1. Unified Analysis UI (trials + region reports)

Merge trial and region-report context; NDVI & yield time series; season-long metadata; in-platform analysis; protocol + ad-hoc regions.

Requirements → UI

NDVI + Yield trends Time-series strip + layer toggle
Season-long region report metadata Metadata section · legacy Region report modal
Protocol vs ad-hoc regions Active regions · map legend · protocol / ad-hoc overlays
No export required Analysis on canvas + chart; export out of scope for this shell

Acceptance criteria

  • Users can view NDVI and Yield trends over time for a selected trial (layer + time range).
  • Users can toggle between NDVI and Yield layers.
  • Users can analyze protocol-defined and ad-hoc regions.
  • Analysis updates dynamically from selected regions and time range (represented by chart + copy).
  • Season-long region report metadata is accessible in the Analysis UI.
  • No dependency on external tools for core analysis workflow (static demo implies in-app).

2. Exclude poor or invalid areas

Exclusions are overlays; raw data unchanged; default = full trial; recalc analysis when exclusions change.

Requirements → UI

Draw / select exclusion Map tools · Exclusions list
Visual differentiation Hatched exclusion · Legend
Raw vs adjusted view Callout

Acceptance criteria

  • Users can draw or select an exclusion zone on the map (tool affordance + list).
  • Excluded areas are visually differentiated from valid analysis area.
  • Analysis recalculates from remaining valid area (implied by “updates with exclusions” on chart).
  • Raw source data remains unchanged and recoverable (callout + reset).
  • Users can remove or edit exclusions (list + draw flow stub).
  • Clear distinction between raw data and adjusted analysis view.
  • Self-service — no engineering step shown in UI.

3. Annotation & performance context

Points, areas, exclusions; structured reasons; summary comment; collaboration; annotations do not auto-change official metrics.

Requirements → UI

Point / area tools Map tools
Reasons (structured) Annotations + reason dropdown
Trial summary comment Summary textarea
Observational vs computed Callout + annotation intro copy

Acceptance criteria

  • Point and area annotations supported (tools + example rows).
  • Annotations can tie to exclusion context (workflow copy in exclusions + annotations).
  • Each annotation supports reason and comment (reason list + sample text).
  • High-level trial summary comment field present.
  • Visibility / collaboration / audit trail — product backend; UI reserves summary + list pattern.
  • Annotations do not alter official processed values automatically (stated in sidebar).
  • UI differentiates observational notes vs computed analysis.

4. System behavior & constraints

Separation of raw data, analytical transforms, and user overlays; performance TBD; iterative analysis without data loss.

Requirements → UI

State separation Raw vs analysis callout
Reset to default Reset analysis
Multiple overlays Map shows region + exclusion simultaneously

Acceptance criteria

  • Switching analysis state does not imply mutation of underlying data (copy + reset).
  • Users can reset analysis to default (no exclusions / default overlay state) via control.
  • Multiple exclusions and annotations supported in layout (stacked list + map).
  • Latency threshold TBD — not shown in static HTML.
